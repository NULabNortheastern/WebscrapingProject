This dictionary defines the most used GitHub terminology.  

  • Repository: a managed collection of files representing code, documents, and digital objects.
  • Fetch: to request information about changes made to the main repository (not yet represented in the local repository).
  • Pull: to actually pull in the changes from the main repository into the local repository.
  • Commit: to formally register a set of local changes so that the local version control is aware of them. A “commit” is a group of such changes that are all registered at the same time.
  • Commit message: a brief description of the changes made during a given “commit.”
  • Push: to transfer a set of changes (a “commit”) to the main repository so that they are public and available to other users of the repository.
  • Version: a formal, recoverable state of the repository associated with a specific set of changes.
  • Clone: a duplicate of an entire repository, including its version history.
  • Branch: a parallel version of a repository (often used as a space to create a new feature without disrupting work that depends on the main branch).
  • Merge: reintegrating the code from one branch with the code from another branch (for instance, to add a feature that was developed in an experimental branch).
  • Fork: a copy of an entire repository; a “fork” is basically a clone that is intended to serve as the starting point for a divergent or independent project.
  • Pull request: a notification that changes made in a branch or fork are ready to be integrated into another branch (hence, a request for that other branch to “pull in” those changes).
